# Systematic Debugging (체계적 디버깅)

## Iron Law
```
근본 원인 조사 없이 수정 금지.
```

랜덤 수정은 시간 낭비이고 새 버그를 만든다.

## 4단계 프로세스

### Phase 1: 근본 원인 조사 (수정 시도 전 필수)

1. **에러 메시지 정독**
   - 스택 트레이스 전체 읽기
   - 라인 번호, 파일 경로, 에러 코드 기록
   - 에러 메시지를 건너뛰지 마라

2. **재현**
   - 안정적으로 재현 가능한가?
   - 정확한 재현 단계는?
   - 재현 불가능하면 → 추측 금지, 데이터 더 수집

3. **최근 변경 확인**
   - `git diff`, 최근 커밋
   - 새 의존성, 설정 변경
   - 환경 차이

4. **데이터 흐름 추적**
   - 잘못된 값의 출발점은 어디?
   - 누가 이 값을 넘겼는가?
   - 원인(source)에서 수정, 증상(symptom)에서 수정 금지

### Phase 2: 패턴 분석

1. **작동하는 유사 코드 찾기** — 같은 코드베이스에서
2. **차이점 나열** — 작동 vs 안 작동 비교
3. **"그건 상관없어"라고 가정하지 마라** — 작은 차이도 기록

### Phase 3: 가설 검증

1. **단일 가설 수립** — "X가 근본 원인이다. 왜냐하면 Y"
2. **최소 변경으로 검증** — 한 번에 하나만 변경
3. **여러 수정을 동시에 하지 마라** — 뭐가 작동했는지 알 수 없다

### Phase 4: 수정 구현

1. **실패하는 테스트 케이스 먼저** (TDD 스킬 참조)
2. **단일 수정** — 근본 원인만 수정
3. **검증** — 테스트 통과? 다른 테스트 안 깨짐?
4. **3번 이상 수정 실패 시 → STOP**
   - 아키텍처 문제일 수 있다
   - 매니저(이서)에게 보고 후 방향 논의

## Red Flags — 즉시 멈추고 Phase 1로 복귀

- "일단 X 바꿔보자"
- "빠르게 수정하고 나중에 조사하자"
- "아마 X일 거야"
- 3번 이상 수정 시도 후에도 해결 안 됨
- 수정할 때마다 다른 곳에서 새 문제 발생

## 합리화 차단

| 변명 | 현실 |
|------|------|
| "간단한 문제야, 프로세스 필요 없어" | 간단한 문제도 근본 원인 있다 |
| "급해, 시간 없어" | 체계적 디버깅이 추측보다 빠르다 |
| "일단 수정부터, 조사는 나중에" | 첫 수정이 패턴을 결정한다. 처음부터 제대로 |
| "이미 3시간 봤어..." | 매몰비용. 멈추고 아키텍처 재검토 |
