# Verification Before Completion (완료 전 검증)

## Iron Law
```
검증 증거 없이 완료 주장 금지.
```

검증 없이 "완료"라고 하는 것은 거짓말이다.

## Gate Function

작업 완료를 주장하기 전에 **반드시**:

1. **IDENTIFY**: 이 주장을 증명하는 명령은?
2. **RUN**: 명령을 실행 (이전 결과 재사용 금지)
3. **READ**: 전체 출력 읽기, exit code 확인, 실패 개수 확인
4. **VERIFY**: 출력이 주장을 뒷받침하는가?
   - NO → 실제 상태를 보고
   - YES → 증거와 함께 주장
5. **THEN**: 그때서야 완료 주장 가능

## 주장별 필수 검증

| 주장 | 필수 검증 | 부족한 검증 |
|------|-----------|-------------|
| "테스트 통과" | 테스트 명령 출력: 0 failures | 이전 실행, "통과할 거야" |
| "빌드 성공" | 빌드 명령: exit 0 | 린터 통과, "에러 없어 보여" |
| "버그 수정됨" | 원래 증상 테스트: 통과 | 코드 변경했으니 됐을 거야 |
| "배포 완료" | 서버 URL 접근: 200 OK | scp 완료 |

## 필수 검증 체크리스트 (모든 작업)

```bash
# 1. 빌드 확인
npm run build  # 또는 npx expo export --platform web
# exit code 0 확인

# 2. 테스트 실행
npm test
# 0 failures 확인

# 3. 배포 후 (배포 작업인 경우)
curl -s -o /dev/null -w "%{http_code}" http://서버URL/
# 200 확인
```

## Red Flags — 즉시 멈춤

이런 말을 하려는 순간 → STOP, 검증부터:
- "통과할 거예요"
- "아마 될 거예요"  
- "이것만 하면 끝이에요"
- "완료했습니다" (검증 명령 없이)
- "Great!", "Perfect!", "Done!" (검증 전)

## 합리화 차단

| 변명 | 현실 |
|------|------|
| "이번엔 될 거야" | 검증 명령을 실행하라 |
| "확신해" | 확신 ≠ 증거 |
| "이번 한 번만" | 예외 없음 |
| "린터 통과했으니" | 린터 ≠ 컴파일러 ≠ 테스트 |
| "피곤해서..." | 피곤함 ≠ 변명 |

## 핵심 원칙

**증거가 주장보다 먼저.**

명령을 실행하라. 출력을 읽어라. **그 다음에** 결과를 주장하라.

이것은 협상 불가.
