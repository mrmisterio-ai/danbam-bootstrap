# Test-Driven Development (TDD)

## Iron Law
```
테스트 없는 코드 작성 금지. 코드를 먼저 작성했다면 삭제하고 다시 시작.
```

사장님 지시: **모든 새 기능은 테스트 먼저 작성 후 코드 작성 (Red → Green → Refactor)**

## Red-Green-Refactor 사이클

### 1. RED — 실패하는 테스트 작성
- 하나의 동작만 테스트
- 명확한 테스트 이름 (행동 설명)
- 실제 코드 사용 (mock은 불가피한 경우만)

```typescript
test('빈 이메일이면 에러 반환', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('Email required');
});
```

### 2. RED 확인 — 실패 확인 필수
```bash
npm test -- --testPathPattern="path/to/test"
```
- 테스트가 **실패**하는지 확인 (에러가 아닌 실패)
- 실패 메시지가 예상과 일치하는지 확인
- **테스트가 통과하면?** → 이미 존재하는 동작을 테스트 중. 테스트 수정.

### 3. GREEN — 최소한의 코드 작성
- 테스트를 통과시키는 **가장 간단한** 코드
- 기능 추가 금지, 리팩토링 금지, "개선" 금지

### 4. GREEN 확인 — 통과 확인 필수
```bash
npm test
```
- 새 테스트 통과?
- 기존 테스트도 통과?
- 경고/에러 없는 깨끗한 출력?

### 5. REFACTOR — 정리
- GREEN 상태에서만 리팩토링
- 중복 제거, 이름 개선, 헬퍼 추출
- 테스트는 계속 GREEN 유지
- 새로운 동작 추가 금지

## 합리화 차단

| 변명 | 현실 |
|------|------|
| "너무 간단해서 테스트 필요 없어" | 간단한 코드도 깨진다. 테스트 30초면 된다. |
| "나중에 테스트 추가할게" | 나중에 추가한 테스트는 아무것도 증명 안 한다. |
| "이미 수동 테스트 했어" | 수동 테스트는 기록 없고, 재실행 불가. |
| "기존 코드 참고만 할게" | 참고하는 순간 적응하게 된다. 삭제하고 다시 시작. |
| "TDD가 느려" | TDD가 디버깅보다 빠르다. |

## 작업 완료 체크리스트

- [ ] 모든 새 함수/메서드에 테스트 있음
- [ ] 각 테스트가 실패하는 것을 확인함
- [ ] 각 테스트가 올바른 이유로 실패함
- [ ] 최소한의 코드로 테스트 통과시킴
- [ ] 전체 테스트 통과
- [ ] 깨끗한 출력 (에러, 경고 없음)
- [ ] 엣지 케이스와 에러 케이스 커버됨
