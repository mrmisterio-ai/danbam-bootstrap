# Design Review — 레퍼런스 기반 + 크리틱 에이전트 루프

## 핵심 원칙
1. **레퍼런스 없이 디자인하지 않는다.**
2. **이서가 직접 크리틱하지 않는다** — 오수아(design-critic)에게 spawn.
3. **크리틱 Pass 없이 사장님에게 전달하지 않는다.**

## 디자인 작업 파이프라인

### Phase 1: 리서치 & 레퍼런스 (이서 준비)
1. web_search로 경쟁앱/유사앱 최소 3개 URL 수집
2. 디자인 요구사항 정리 (색상 규칙, 레이아웃, 컴포넌트)
3. 레퍼런스 URL을 토니 태스크에 포함

### Phase 2: 디자인 제작 (토니)
토니 태스크에 필수 포함:
- 레퍼런스 URL 3개 이상
- 60-30-10 색상 법칙 명시
- Atomic Design 구조 (atoms → molecules → organisms → templates → pages)
- AC에 "키 컬러 15% 이하" 포함

### Phase 3: 크리틱 루프 (오수아 — design-critic)
토니 완료 후, **이서가 직접 보지 않고** 오수아에게 크리틱 spawn:

```
sessions_spawn({
  agentId: "design-critic",
  task: "토니가 만든 [URL]을 크리틱해줘. 레퍼런스: [URLs]. 7항목 체크리스트로 평가.",
  label: "design-critique-[작업명]"
})
```

오수아 결과:
- **Pass** → Phase 4로
- **Conditional Pass** → 마이너 수정 사항을 토니에게 전달 → 수정 → 재크리틱
- **Fail** → 메이저 수정 사항을 토니에게 전달 → 재작업 → 재크리틱

**최소 1회 크리틱 필수. 첫 제출에서 Pass하더라도 기록 남김.**

### Phase 4: 사장님 보고 (이서)
크리틱 Pass 결과 + 프리뷰 URL을 사장님에게 전달.
크리틱 리포트 요약 포함.

## 이서의 역할 (최소화)
- 레퍼런스 URL 수집 + 태스크 작성 (Phase 1)
- 토니/오수아 spawn 및 결과 중계 (Phase 2~3)
- 사장님에게 종합 보고 (Phase 4)
- **직접 스크린샷 캡처/비교하지 않는다** — 오수아가 함

## 60-30-10 색상 법칙
- 60% 중립 — 배경, 카드 배경
- 30% 보조 — 보더, 비활성 아이콘, 서브텍스트
- 10% 키 컬러 — CTA, 활성 탭, 핵심 강조만
- **키 컬러 15% 초과 = 즉시 Fail**

## ⚠️ 절대 규칙
1. 레퍼런스 없이 토니 spawn 금지
2. 크리틱(오수아) 없이 사장님 전달 금지
3. 이서가 직접 디자인 판단하지 않음 — 오수아에게 위임
4. "코드로 그린 SVG" 허용 안 함 — 이미지 필요하면 생성 도구 사용
